# Quickstart

This walkthrough wires up one lane end to end: define a message, produce it
inside your own database transaction, run the schema setup, relay it over
AMQP, consume it with deduplication and retry, and register the two
long-running processes as CLI commands.

It assumes a MySQL or PostgreSQL database and a RabbitMQ broker reachable
from your application. See [TRANSPORTS.md](TRANSPORTS.md) for the Kafka
transport and the AMQP competing-relay mode, and
[CLI_REFERENCE.md](CLI_REFERENCE.md) for every command's full option list.

## Define a message

Userland messages extend `Message` and expose a named constructor that
builds the payload explicitly — there is no reflection-based normalization.
`id` and `createdAt` are generated by the base class; the payload is
whatever your named constructor puts in the array.

```php
use Freyr\MessageBroker\Message;

final class OrderPlaced extends Message
{
    public static function create(string $orderId, int $totalCents): self
    {
        return new self(key: $orderId, name: 'order.placed', payload: [
            'order_id' => $orderId,
            'total_cents' => $totalCents,
        ]);
    }
}
```

## Produce inside your transaction

`OutboxProducer` writes through your application's own `PDO` connection, so
the outbox row commits or rolls back atomically with your own state change.
`OutboxStore` is the storage interface; `PdoOutboxStore` is the shipped PDO
implementation — construct it with your connection and a `Platform`
(`MySqlPlatform` or `PostgreSqlPlatform`).

The wire format is encoded at produce time: `JsonWireFormat` (shown here) or
`AvroWireFormat` (see [SCHEMA_AVRO.md](SCHEMA_AVRO.md)) turns the payload
into the exact bytes stored in the outbox `body` column, and a
non-conforming payload throws inside your transaction — nothing commits.

```php
use Freyr\MessageBroker\Outbox\OutboxProducer;
use Freyr\MessageBroker\Outbox\PdoOutboxStore;
use Freyr\MessageBroker\Serializer\JsonWireFormat;
use Freyr\MessageBroker\Storage\MySqlPlatform;

$pdo = new PDO($dsn, $user, $password, [PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION]);
$outbox = new PdoOutboxStore($pdo, new MySqlPlatform());
$producer = new OutboxProducer($outbox, new JsonWireFormat(), lane: 'orders');

$pdo->beginTransaction();
// ... your own state change ...
$producer->produce(OrderPlaced::create('o-1', 4999), headers: ['correlation_id' => 'c-1']);
$pdo->commit(); // the outbox row commits atomically with your data
```

A *lane* is a named drain of the outbox table. One producer instance writes
to one lane; exactly one ordered relay serves that lane at a time, which is
what guarantees total in-order publishing per lane.

## Run the schema setup

Before producing anything, create `outbox_messages`, `message_deduplication`,
and `dead_letters`:

```bash
php bin/console message-broker:setup:schema --format=json
```

`--format` is a global, setup-time choice (`json` or `avro`) — it picks the
`body` column type (JSON text vs. `LONGBLOB`/`BYTEA`) and must match the
`WireFormat` your producer uses. See [CLI_REFERENCE.md](CLI_REFERENCE.md) for
`--dump-sql` and every other command's options.

## Run a relay

A relay drains one outbox lane to one transport target, preserving order. It
needs a dedicated `AMQPChannel` — publisher confirms are channel-global
state, so nothing else may publish on the same channel.

```php
use Freyr\MessageBroker\Transport\Amqp\AmqpPublishConfig;
use Freyr\MessageBroker\Transport\Amqp\AmqpRelay;

$relay = new AmqpRelay(
    outbox: $outbox,
    amqp: $channel,                                  // a dedicated PhpAmqpLib AMQPChannel
    publish: new AmqpPublishConfig(exchange: 'orders'),
    contentType: JsonWireFormat::CONTENT_TYPE,
    lane: 'orders',
);
$relay->run(); // blocks; SIGTERM/SIGINT to stop
```

This is the *ordered* relay — the default, and the one to reach for unless
your consumers are explicitly order-insensitive. See
[TRANSPORTS.md](TRANSPORTS.md) for the opt-in `CompetingAmqpRelay` and for
the Kafka relay.

## Run a consumer

A consumer deserializes each delivery, deduplicates and dispatches inside
one database transaction, then acks. Failures go through a per-transport
retry policy and land in the shared `dead_letters` table once the attempt
budget is exhausted.

```php
use Freyr\MessageBroker\Consumer\CallableDispatcher;
use Freyr\MessageBroker\Consumer\IncomingMessage;
use Freyr\MessageBroker\DeadLetter\PdoDeadLetterStore;
use Freyr\MessageBroker\Retry\Backoff;
use Freyr\MessageBroker\Serializer\JsonDeserializer;
use Freyr\MessageBroker\Transport\Amqp\AmqpConsumer;
use Freyr\MessageBroker\Transport\Amqp\AmqpQueueConfig;
use Freyr\MessageBroker\Transport\Amqp\AmqpRetryPolicy;
use Freyr\MessageBroker\Transport\PdoDeduplicationStore;

$consumer = new AmqpConsumer(
    channel: $channel,
    queue: new AmqpQueueConfig('orders_q'),
    deserializer: new JsonDeserializer(),
    dispatcher: new CallableDispatcher(static function (IncomingMessage $m): void {
        // your handler — denormalize $m->payload and act
    }),
    pdo: $pdo,
    deduplication: new PdoDeduplicationStore($pdo, new MySqlPlatform()),
    retryPolicy: new AmqpRetryPolicy(maxAttempts: 5, backoff: Backoff::exponential(initialDelayMs: 1_000, maxDelayMs: 300_000)),
    deadLetters: new PdoDeadLetterStore($pdo, new MySqlPlatform()),
    name: 'orders_consumer',
);
$consumer->run(); // blocks
```

`name` scopes deduplication — the same message id delivered to two
differently-named consumers is processed by both; delivered twice to the
same named consumer is processed once.

## Register the CLI

Relays and consumers are long-running processes. Register each one as a
named callable on `RelayRunCommand`/`ConsumeCommand` so an operator (or a
supervisor) starts it by lane/consumer name:

```php
use Freyr\MessageBroker\Console\ConsumeCommand;
use Freyr\MessageBroker\Console\RelayRunCommand;

$application->add(new RelayRunCommand(['orders' => static fn () => $relay->run()]));
$application->add(new ConsumeCommand(['orders' => static fn () => $consumer->run()]));
```

Run them with `php bin/console message-broker:relay:run orders` and
`php bin/console message-broker:consume orders`. Deploy both under a process
supervisor with restart-always — see [PRODUCTION.md](PRODUCTION.md) for the
crash-only deployment model and why it is safe.
